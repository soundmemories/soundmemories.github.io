数据结构概述

不管是哪一种数据结构，其实本质上都是容器来着，就是用来装对象的。因此就要搞清楚两点：（1）如何存储（2）存储特点。

数据结构:
（1）逻辑结构：在物理结构基础上，进行操作结构。同一种逻辑结构可以有多种物理结构，比如树可以用顺序存储，也可以用链式存储。
（2）物理结构：就是存储结构，包括数据元素的表示和关系的表示。

逻辑结构：
（1）线性结构：动态数组、栈、队列
　　数据元素之间存在着“一对一”的线性关系。
（2）非线性结构：二叉树、图
　　数据元素之间存在着“一对多”的非线性关系，可细分为树形和图形。

物理结构（存储结构）：
（1）顺序存储：连续的存储空间
　　元素是相邻的，在内存中需要开辟的连续的存储空间。
　　优点：访问速度比较快，因为有了首地址，然后根据下标，直接就可以找到对应的元素。
　　缺点：内存比较吃紧时，找一整块空间不好找。
（2）链式存储：非连续的存储空间
　　元素不一定是相邻的，在内存中不需要开辟的连续的存储空间。
　　优点：不要求空间连续，可以插空。
　　缺点：访问速度相对数组比较慢，要从头，挨个遍历。
物理结构：顺序表、链表等。

数组

（1）逻辑结构：线性的。
（2）物理结构：顺序的存储结构。
（3）申请内存：一次申请一大段连续的空间，一旦申请到了，内存就固定了。
（4）存储特点：所有数据存储在连续的空间中，数组中的每一个元素都是一个具体的数据（或对象），所有数据都紧密排布，不能有间隔。

数组操作：
（1）查询：每一个元素都有一个数值下标，可以通过下标瞬间定位到某个元素。
（2）增加：先使用total变量辅助记录实际存储元素个数，从尾部增加（数组名[total++]=新元素;），从其他位置插入（先把index位置开始所有元素后移，然后数组名[total++]=新元素;）。
（3）删除：先把index后面的元素前移，然后数组名[total--]=null;。
（4）改：直接数组名[index]=新元素;。

优缺点：
（1）优点：按索引查询效率高。
（2）缺点：添加/删除效率低，因为都涉及到移动元素；无法直接获取有效元素的个数，需要total来辅助；长度是固定的。

动态数组：
（1）物理结构：数组。
（2）逻辑结构：动态数组（对于使用者来说，可以自动扩容，自动记录有效元素的个数等功能）。
MyArrayList就是自定义的物理结构，它有增删改查等功能的具体实现。
TestMyArrayList就是逻辑逻辑结构，它是动态数组类型的，因为底层的物理结构是基于数组的。

集合

系统预定义的逻辑结构。物理结构是装对象的，并没有对物理结构的逻辑操作，比如增删改查这种逻辑操作，所以为了可以满足用户数据更多种的逻辑关系，而设计的一系列的不同于数组的可变的聚合的抽象数据类型，来抽象出这种逻辑操作。
这些接口和类在java.util包中，因为类型很丰富，因此我们通常称为集合框架集，主要分为Collection和Map两大类接口。
Collection：规范单值集合的接口，obj。
Map：规范对值集合的接口，(key,value)。

Collection集合

（1）它是是其他单值集合的根接口。
（2）它没有直接的实现类，而是由具体的子接口（List、Set、Queue等）中实现。
（3）一些集合允许有重复的元素，而另一些则不允许。一些集合是有序的，而另一些则是无序的。

方法

1、添加元素
（1）add(Object obj)：添加元素对象到当前集合中。
（2）addAll(Collection other)：添加other集合中的所有元素对象到当前集合中，即this = this ∪ other。
2、删除元素
（1）boolean remove(Object obj)：从当前集合中删除第一个找到的与obj对象equals返回true的元素。
（2）boolean removeAll(Collection coll)：从当前集合中删除所有与coll集合中相同的元素。即this = this - this ∩ coll。
3、判断元素
（1）boolean isEmpty()：判断当前集合是否为空集合，等价于if(集合对象.size()==0)。
（2）boolean contains(Object obj)：判断当前集合中是否存在一个与obj对象equals返回true的元素。
（3）boolean containsAll(Collection c)：判断c集合中的元素是否在当前集合中都存在。即c集合是否是当前集合的“子集”。
4、其他
（1）int size()：获取当前集合中实际存储的元素个数。
（2）void clear()：清空当前集合。
（3）Object[] toArray()：返回包含当前集合中所有元素的数组。
（4）Iterator iterator()：返回当前集合的迭代器对象。
（5）boolean retainAll(Collection coll)：当前集合仅保留与c集合中的元素相同的元素(两个集合的交集)，即this = this ∩ coll。

遍历

1、Iterator迭代器遍历
因为Collection接口继承了java.lang.Iterable接口，该接口有一个iterator()方法，那么所有实现了Collection接口的集合类都有一个iterator()方法，该方法返回一个实现了Iterator接口的对象，该对象可以用于迭代集合中的元素。

java.util.Iterator：迭代器接口，这个接口的实现类在每一种集合类中，例如：ArrayList内部有一个内部类实现了Iterator接口。
声明为内部类原因：
（1）每一种集合的内部实现（物理结构不同），意味着对迭代器（售票员）的实现是不同的，每一种集合都要单独定制迭代器。
（2）内部类可以直接访问外部类的私有的属性，成员，迭代器就可以直接访问集合的私有的元素。


  • Iterator 仅用于遍历集合，Iterator 本身并不提供承装对象的能力。如果需要创建 Iterator 对象，则必须有一个被迭代的集合。
  • 集合对象每次调用iterator()方法都得到一个全新的迭代器对象，默认游标都在集合的第一个元素位置。

常用方法：
（1）boolean hasNext()：如果仍有元素可以迭代，则返回 true。
（2）Object next()：返回迭代的下一个元素。
（3）void remove()：从迭代器指向的 collection 中移除迭代器返回的最后一个元素（可选操作）。
  • 在调用it.next()方法之前必须要调用it.hasNext()进行检测。若不调用，且下一条记录无效，直接调用it.next()会抛出NoSuchElementException异常。
  • 如果还未调用next()或在上一次调用 next 方法之后已经调用了 remove 方法，再调用remove都会报IllegalStateException。

2、foreach循环遍历
Java 5时Collection接口继承了java.lang.Iterable接口，因此Collection系列的集合就可以直接使用foreach循环遍历。


foreach遍历只适用于查看/查找集合中的元素，不能在遍历集合时有任何影响集合元素个数的操作，否则报异常或者操作结果将不确定。
因为使用foreach，删除时调用的是集合的remove方法，遍历调用的是Iterator的遍历方法，这是两个路线，那么就导致结果不一致。集合为了避免这种错误，设计java.util.ConcurrentModificationException异常，有的时候可能不报异常，所以避免使用foreach删除。

提示：集合中只能添加对象，不能添加基本数据类型的数据，如果添加基本数据类型的数据，它会自动装箱为对应包装类型的对象。

凡是实现了java.lang.Iterable接口（可迭代）的集合或容器都支持foreach循环。
foreach底层还是调用Iterator迭代器来遍历集合。
自定义容器实现iterator参考TestIterable。

  • TestForeach
  • TestIterable

思考：如果遍历数组，什么情况下选用foreach，什么情况下选用for循环？
当如果你的操作中涉及到[下标]操作时，用for最好。
当你只是查看元素的内容，那么选foreach更简洁一些。

思考：如果遍历Collection系列集合，什么情况下选用foreach，是否能选用for循环？
首先考虑使用foreach，如果该集合也有索引信息的话，也可以通过for来操作，如果没有下标的信息，就不要用for。即，如果该集合的物理结构是数组的，那么可以用for，如果物理结构是链式，那么使用下标操作效率很低。

思考：如果遍历Collection系列集合，什么情况下选用foreach，什么情况下使用Iterator？
如果只是查看集合的元素，使用foreach，代码会更简洁。
但是如果要涉及到在遍历集合的同时根据某种条件要删除元素等操作，那么选用Iterator。

List集合

List集合类中元素有序、且可重复。
List系列常用集合：Vector（动态数组），ArrayList（动态数组），Stack（栈），LinkedList（栈，双向链表，双端队列(JDK1.6后)）。

  • 在JavaSE中List名称的类型有两个，一个是java.util.List集合接口，一个是java.awt.List图形界面的组件，别导错包了。
  • List集合关心元素是否有序，而不关心是否重复，请大家记住这个原则。例如“张三”可以领取两个号。

方法

List除了从Collection集合继承的方法外，List 集合里添加了一些根据“索引”来操作集合元素的方法。
1、添加元素
（1）void add(int index, Object ele)：在[index]位置添加一个对象，相当于插入。
（2）boolean addAll(int index, Collection eles)：在[index]位置添加eles中的所有元素。
2、获取元素
（1）Object get(int index)：获取，返回[index]位置的元素。
（2）List subList(int fromIndex, int toIndex)：截取一部分，[fromIndex, toIndex)。
3、获取元素索引
（1）int indexOf(Object obj)：在当前集合中正序查找第一个obj的下标，如果没有返回-1。
（2）int lastIndexOf(Object obj)：在当前集合中倒序查找第一个obj的下标，如果没有返回-1。
4、删除和替换元素
（1）Object remove(int index)：删除指定位置的元素。
（2）Object set(int index, Object ele)：替换[index]位置的元素。
5、其他
（1）ListIterator listIterator() ：返回一个迭代器。

虽然List系列的集合都是可以通过[索引/下标]进行操作的，但是像LinkedList这类的集合，其实不建议使用和索引相关的方法进行操作。因为它们的底层的物理结构不是数组，如果通过索引操作呢，会需要从头或尾遍历找到对应的索引，效率不高。


实现类

1、Vector：动态数组（旧版），物理结构是数组。
2、ArrayList：动态数组（新版），物理结构是数组。
3、Stack：栈，它是Vector的子类，物理结构是数组。
4、LinkedList：双向链表，双端队列(JDK1.6后)，物理结构是链表。

ArrayList与Vector的区别与联系？
（1）它们的底层物理结构都是数组，我们称为动态数组。
（2）它们初始化容量可用initialCapacity指定，增量可用capacityIncrement指定。
（4）Vector：最早版本的动态数组（旧版）。
　　Vector是线程安全的，效率低；初始容量是10，不够后扩容为原来的2倍，缺点是造成空间浪费的可能性比较大。
　　Vector支持的遍历集合的方式有：（1）foreach（2）Iterator（3）支持旧版的Enumeration迭代器。
（4）ArrayList：相对Vector来说新一点（新版）。
　　ArrayList是线程不安全的，效率高；1.6初始容量是10，之后默认为空数组，不够后扩容为原来的1.5倍，缺点是造成扩容的次数增大。
　　ArrayList支持的遍历集合的方式有：（1）foreach（2）Iterator。

Vector因为版本古老，支持Enumeration 迭代器。但是该迭代器不支持快速失败。而Iterator和ListIterator迭代器支持快速失败。如果在迭代器创建后的任意时间从结构上修改了向量（通过迭代器自身的 remove 或 add 方法之外的任何其他方式），则迭代器将抛出 ConcurrentModificationException。因此，面对并发的修改，迭代器很快就完全失败，而不是冒着在将来不确定的时间任意发生不确定行为的风险。

Vector与Stack的区别与联系？
（1）Stack类是Vector的子类。
（2）Stack类表示后进先出（LIFO）的对象堆栈。它通过五个操作对类 Vector 进行了扩展，允许将向量视为堆栈：
Object peek()：访问当前栈顶元素，但是不拿走栈顶元素。
Object pop()：弹出栈顶元素。
Object push(Object item)：把元素压入栈顶，等价于add(item)，但比其可读性更好些。
boolean empty()：测试堆栈是否为空。
int search(Object o)：在堆栈中查找项并确定到堆栈顶距离，以1为基数。

Set集合

Set接口是Collection的子接口，set接口没有提供额外的方法。因此Set集合支持的遍历方式和Collection集合一样：foreach和Iterator。
Set 集合不允许包含相同的元素，如果试把两个相同的元素加入同一个 Set 集合中，则添加操作失败。
Set的常用实现类有：
（1）HashSet：既不能保证添加顺序，也不能保证大小顺序，是完全无序的。
（2）TreeSet：唯有TreeSet可以保证元素的大小顺序。
（3）LinkedHashSet：唯有LinkedHashSet可以保证元素添加的顺序。

HashSet和LinkedHashSet：
（1）LinkedHashSet是HashSet的子类，它在HashSet的基础上，在结点中增加两个属性before和after维护了结点的前后添加顺序。
（2）HashSet 和LinkedHashSet按 Hash 算法来存储集合中的元素，因此具有很好的存取和查找性能。插入 HashSet 性能更好，迭代访问LinkedHashSet更好。
（3）HashSet 和LinkedHashSet集合判断两个元素相等的标准：两个对象先通过 hashCode() 方法比较hash值，如果hash不一样，说明一定不相同，如果hash值一样，再调用equals方法比较。因此，存储到HashSet和LinkedHashSet的元素要重写hashCode和equals方法。重写equals()方法的原则：根中的主要方法equals(Object obj)。

TreeSet：
TreeSet 是 SortedSet 接口的实现类，TreeSet 可以确保集合元素处于排序状态。
当需要元素不可重复，又要给元素排大小时，就用TreeSet。
要用到TreeSet，元素对象一定要实现java.lang.Comparable（自然排序）或java.util.Comparator（定制排序），当其返回0时，就认为相等，不会使用hashCode和equals方法判断相等。

虽然添加到TreeSet时，不使用equals方法。但当元素实现java.lang.Comparable接口时，重写compareTo方法时，也建议重写 equals() 方法，应保证该方法与 compareTo(Object obj) 方法有一致的结果：如果两个对象通过 equals() 方法比较返回 true，则通过 compareTo(Object obj) 方法比较应返回 0。否则让人难以理解。

Set的源码分析:
Set的内部实现其实是一个Map。
HashSet的内部实现是一个HashMap。
TreeSet的内部实现是一个TreeMap。
LinkedHashSet的内部实现是一个LinkedHashMap。

Map集合

（1）Map 中的 key 不允许重复。
（2）Map 中的 key 和 value 都可以是任何引用类型的数据，遇到最多的key的类型是：String和Integer。因为比较简洁，而且对象不可变。
（3）key 和 value 之间存在单向一对一关系，即通过指定的 key 总能找到唯一的、确定的 value。
（4）Map 中的key-value的类型：Entry类型。

方法

1、添加操作
（1）Object put(Object key,Object value)：将指定的值与此映射中的指定键关联。
（2）void putAll(Map t)：从指定映射中将所有映射关系复制到此映射中。
2、删除
（1）void clear()：从此映射中移除所有映射关系。
（2）Object remove(Object key)：如果存在一个键的映射关系，则将其从此映射中移除。
3、元素查询的操作
（1）Object get(Object key)：返回指定键所映射的值；如果此映射不包含该键的映射关系，则返回 null。
（2）boolean containsKey(Object key)：如果此映射包含指定键的映射关系，则返回 true。
（3）boolean containsValue(Object value)：如果此映射将一个或多个键映射到指定值，则返回 true。
（4）boolean isEmpty()：如果此映射未包含键-值映射关系，则返回 true。
4、其他方法
（1）int size()：返回此映射中的键-值映射关系数。

遍历

Map接口没有继承java.lang.Iterable接口，所以不支持直接使用foreach循环进行遍历。
Map接口中也没有提供Iterator iterator()方法返回迭代器对象。

Map的遍历：
（1）单独遍历所有key：Set keySet()返回此映射中包含的键的 Set 视图。
（2）单独遍历所有value：Collection values()返回此映射中包含的值的 Collection 视图。
（3）成对遍历key-value：Set entrySet()返回此映射中包含的映射关系的 Set 视图。

Map.Entry是Map接口的内部接口。每一种Map内部有自己的Map.Entry的实现类。
在Map中存储数据，实际上是将Key-value的数据存储在Map.Entry接口的实例中，再在Map集合中插入Map.Entry的实例化对象。


实现类

1、HashMap
2、Hashtable
3、TreeMap
4、LinkedHashMap
5、Properties

所有的map的key不能重复，如何实现不重复?
HashMap、Hashtable、LinkedHashMap、Properties：依据key的hashCode和equals方法。
TreeMap：依据key的自然顺序大小，认为大小相等的两个key就是重复的，只与Comparable和Comparator有关。
如果key重复了，新的value会替换旧的value。

HashMap和Hashtable：
（1）HashMap和Hashtable都是哈希表。
（2）Hashtable：旧版。线程安全的。它的key和value不能为null。
（3）HashMap：新版。线程不安全。它允许key和value为null值。。
（4）HashMap和Hashtable判断两个 key 相等的标准是：两个 key 的 hashCode 值相等，并且 equals() 方法也返回 true。因此，为了成功地在哈希表中存储和获取对象，用作键的对象必须实现 hashCode 方法和 equals 方法。

类似的还有：StringBuffer和StringBuilder、Vector和ArrayList。

LinkedHashMap：
LinkedHashMap 是 HashMap 的子类，比HashMap多维护了映射关系的添加顺序。

TreeMap：
基于红黑树（Red-Black tree）的 NavigableMap 实现。
该映射根据其key的自然顺序进行排序，或者根据创建映射时提供的 Comparator 进行排序，具体取决于使用的构造方法。
TreeMap要让key排大小，要么key类型本身实现了java.lang.Comparable接口，要么在创建TreeMap时，指定一个java.util.Comparator接口的实现类对象。

Properties：
Properties是Hashtable的子类，它的key和value的类型都是String，不允许key和value是null。
Properties通常用于存储配置属性，而且为了可读性更好，还增加了两个方法：setProperty(String key,String value)方法和getProperty(String key)方法。

源码跟踪

Map的底层实现是什么？
Hash系列：（1）数组 + 链表（2）数组 + 链表/红黑树。
TreeMap：红黑树。

HashMap的底层实现：
JDK1.7以及之前：数组 + 链表。
JDK1.8以及之后：数组 + 链表/红黑树。
数组的优点：访问速度快，因为可以根据下标直接定位到某个元素。
链表的优点：不需要元素是挨着存储，不需要连续空间，在添加和删除元素时不需要移动元素，只需要修改前后元素的引用关系就可以。
（1）添加的映射关系(key,value)最终被封装为一个Map.Entry类型的对象，放到“数组[index]”中。使用数组的目的是查询和添加的效率高。
（2）HashMap会根据key的hashCode+公式/算法，得到[index]。但不同的hashCode值，可能得到的[index]是相同的，这就冲突了。
（3）冲突解决：旧版HashMap，把[index]相同的多个映射关系用链表连接起来，会导致链表很长查询速度慢。新版HashMap，在链表很长时，使用二叉树提高查询效率。

JDK1.7的HashMap：
1、new HashMap()：table数组初始化为了一个长度为0的空数组，threshold=12。
2、put()的存储过程：：
（1）发现数组table是空数组后，会把数组初始化为长度为16的Entry类型的数组，并且把threshold计算为12。这里如果你手动指定了数组的capacity，如果它不是2的n次方，会自动纠正为2的n次方。为什么要纠正为2的次方？后面算index = hash & table.length-1，这样才能保证[0,table.length-1]范围内。2的次方，根据它的散列算法，可以保证比较均匀的分散在它的数组的各个位置。
（2）如果key是null，都放入table[0]中，但table[0]中不只存key为null的对象。
（3）hash = hash(key)为了干扰我们key的hashCode值。index = hash & table.length-1计算index。
（4）先判断table[index]下面是否有某个Entry的key与新的key的“相同”（hash值相同&&(满足key的地址相同||key的equals返回true)），如果有，就用新的value替换旧的value，就结束了。如果没有重复的，决定添加新的映射关系。
（5）添加新的映射关系：先看是否需要扩容（扩容条件：size达到阈值threshold && table[index]不是null），如果扩容了，会重新计算hash和index，并调整存储位置。再把新的映射关系new为一个Entry的对象，放到table[index]中，原来table[index]的映射关系作为新的映射关系的next连接起来。
（6）size++，元素个数增加。

JDK1.8的HashMap：
几个常量和变量：
（1）DEFAULT_INITIAL_CAPACITY：默认的初始容量 16。
（2）MAXIMUM_CAPACITY：最大容量 1 << 30。
（3）DEFAULT_LOAD_FACTOR：默认加载因子 0.75。
（4）TREEIFY_THRESHOLD：默认树化阈值8，当链表的长度达到这个值后，要考虑树化。
（5）UNTREEIFY_THRESHOLD：默认反树化阈值6，当树中的结点的个数达到这个阈值后，要考虑变为链表。
（6）MIN_TREEIFY_CAPACITY：最小树化容量64。当链表的结点个数达到8，且table的长度达到64，才会树化，未达到64，会先扩容。
（7）Node\[] table：数组。
（8）size：记录有效映射关系的对数，也是Entry对象的个数。
（9）int threshold：阈值，当size达到阈值时，考虑扩容。
（10）double loadFactor：加载因子，影响扩容的频率。

1、new HashMap()：table数组初始化为了一个长度为0的空数组，threshold=12。
2、put()的存储过程：：
（1）先计算key的hash值，如果key是null，hash值就是0，如果为null，使用(h = key.hashCode()) ^ (h >>> 16)得到hash值；
（2）如果table是空的，先初始化table为长度为16的数组，threshold = 12；
（3）通过hash值计算存储的索引位置index = hash & (table.length-1)；
（4）如果table[index]==null，那么直接创建一个Node结点存储到table[index]中即可；
（5）如果table[index]!=null，并且table[index]是一个TreeNode结点，说明table[index]下是一棵红黑树，如果该树的某个结点的key与新的key“相同”（hash值相同并且(满足key的地址相同或key的equals返回true)），那么用新的value替换原来的value，否则将(key,value)封装为一个TreeNode结点，连接到红黑树中；
（6）如果table[index]不是一个TreeNode结点，说明table[index]下是一个链表，如果该链表中的某个结点的key与新的key“相同”，那么用新的value替换原来的value，否则需要判断table[index]下结点个数，如果没有达到TREEIFY_THRESHOLD(8)个，那么(key,value)将会封装为一个Node结点直接链接到链表尾部；
（7）如果table[index]下结点个数已经达到TREEIFY_THRESHOLD(8)个，那么再判断table.length是否达到MIN_TREEIFY_CAPACITY(64)，如果没达到，那么先扩容，扩容会导致所有元素重新计算index，并调整位置；
（8）如果table[index]下结点个数已经达到TREEIFY_THRESHOLD(8)个并table.length也已经达到MIN_TREEIFY_CAPACITY(64)，那么会将该链表转成一棵自平衡的红黑树，并将结点链接到红黑树中；
（9）如果新增结点而不是替换，那么size++，并且还要重新判断size是否达到threshold阈值，如果达到，还要扩容。

和1.7扩容区别：1.7须同时满足 达到阈值 && table[i]不是null，1.8只要达到阈值即可。

modCount：记录当前集合被修改的次数。在添加、删除这个两个操作都会修改这个变量。

当我们使用迭代器或foreach遍历时，如果你在foreach遍历时，自动调用迭代器的迭代方法，此时在遍历过程中调用了集合的add,remove方法时，modCount就会改变，而迭代器记录的modCount是开始迭代之前的，如果两个不一致，就会报异常，这说明有两个线路（线程）同时操作集合。这种操作有风险，为了保证结果的正确性，避免这样的情况发生，一旦发现modCount与expectedModCount不一致，立即保错。

此类的 iterator 和 listIterator 方法返回的迭代器是快速失败的：在创建迭代器之后，除非通过迭代器自身的 remove 或 add 方法从结构上对列表进行修改，否则在任何时间以任何方式对列表进行修改，迭代器都会抛出 ConcurrentModificationException。因此，面对并发的修改，迭代器很快就会完全失败，而不是冒着在将来某个不确定时间发生任意不确定行为的风险。

Java异常

发表于 2020-02-10 分类于 Java 本文字数： 4k 阅读时长 ≈ 15 分钟

异常简介

异常，又称为例外，是指不可预知的非正常的情况，这些突发的情况的发生是不可预知的，但是如果处理好了，美好的生活不受影响，可以继续，如果处理不好，可能就“挂”了。
Java语言将程序执行中发生的不正常情况称为“异常”，异常是一种对程序运行过程中发生错误时进行通知的机制。如：网络连接失败、用户输入错误或打开文件失败等。

语法错误不属于异常，这种情况编译不通过，无法运行；
逻辑错误不属于异常，这种情况属于BUG，必须修正；

Java的程序如果出现异常，并且在代码中，没有做相应的处理，那么程序会“崩溃”，“挂了”。

异常的分类

Java语言提供了两大类异常：
（1）编译时异常（受检异常Checked）：指程序中必须接受检查和处理的异常，由Exception 类表示。
（2）运行时异常（非受检异常UnChecked）：程序中可以不接受检查和处理，或致命性错误异常，由RuntimeException类或Error类表示。

异常处理机制

Java中异常以对象的形式出现，不同的异常对象中封装了相应的错误信息。
Java中异常的是以堆栈式抛出机制来进行错误通知的。
Java是采用面向对象的方式来处理异常的，处理过程：
（1）抛出异常：在执行一个方法时，如果在某句代码发生异常，则会在该句代码处生成代表该异常的一个对象，停止当前执行路径，并把异常对象提交给JRE，不同的异常对象中封装了相应的错误信息。
（2）捕获异常：JRE得到该异常后，寻找相应的代码来处理该异常。JRE在方法的调用栈中查找，从生成异常的方法开始回溯，直到找到相应的异常处理代码为止。如果没有找到处理代码，那么程序终止运行。

Java如何处理异常？或Java异常处理的过程？（白话版）
（1）当程序运行到某一句时，发生了异常，那么程序会先停下来；
（2）程序会在这句代码处，查看原因，生成一个合理“异常对象”，然后“抛”出；
（3）JVM会检测在这句代码的外围，是否有try..catch结构，可以“捕获”它：
　　如果可以捕获，那么程序再处理完异常后，继续下面的运行，不会崩溃；
　　如果不能捕获，那么会把这个异常继续抛给“上级”，如果“上级”能处理，那么程序从“上级"处理完的代码后面继续运行；
　　如果上级也不能处理，那么继续往上抛，一直到达JVM，那么就“崩溃”。

Throwable

Throwable就是java.lang.Throwable，是异常系列的超父类：
（1）只有它或它子类的对象，才能被JVM或throw语句“抛”出。
（2）只有它或它子类的对象，才能被catch“捕获”。

所有类的根父类是Object。
枚举的公共父类是Enum，根父类仍然是Object。
异常的公共父类是Throwable，根父类仍然是Object。

Throwable分为两大派别：Error和Exception。
Error：错误
（1）一般指严重错误，一般合理的应用程序不应该试图去捕获它。
（2）如果出现这个问题，要么需要升级程序，要么需要升级架构，要么需要升级硬件。
（3）经典代表，java.lang.VirtualMachineError的子类：OutOfMemoryError（堆内存溢出），StackOverflowError（栈内存溢出）。

Exception：异常
一般异常，合理的应用程序应该试图去捕获它，是应该在程序中尽可能的预知并处理异常。
Exception还可以分为两大类：
（1）运行时异常。又称为非受检异常：
　　即RuntimeException或它子类，编译时，编译器是不会提醒你做处理的，只有运行期间，才会发生。
　　运行时异常是不建议用try...catch，因为它发生频率太高，而且一般都是很不应该发生的问题。
　　例如：空指针异常，数组下标越界异常，类型转换异常等，这些异常完全可以避免掉。
　　但是如果实在没有考虑到，也可以通过try...catch处理。
（2）编译时异常。又称为受检异常：
　　除了运行时异常以外的，都是编译时异常。
　　编译时，编译器会强制要求程序员编写处理的代码，如果你不编写，那么就编译不通过。
　　例如：FileNotFoundException，IOException等

常见的异常：
（1）ArrayIndexOutOfBoundsException：数组下标越界异常->下标超过了[0,长度-1]。
（2）NullPointerException：空指针异常->对象.属性、对象.方法、如果对象是null，就会发生空指针异常。
（3）ClassCastException：类型转换异常->在向下转型时，可能会发生。
（4）ArithmeticException：算术异常->分母为零。
（5）InputMisMatchException：输入类型不匹配异常。
（6）IllegalAugmentException：非法参数异常。

try...catch...finally

1、语法格式：


2、执行特点：
（1）如果try中的代码没有异常，那么try中的代码会正常执行，catch部分就不执行，finally中会执行。
（2）如果try中的代码有异常，那么try中发生异常的代码的后面就不执行了，找对应的匹配的catch分支执行，finally中会执行。

3、多个catch分支，如何匹配和执行的？
从上到下依次判断，一旦有一个满足，后面就不看了。
如果多个catch中的异常类型有大小包含关系，那么小的在上，大的在下，如果没有大小包含关系，顺序随意。

4、finally与return混用：
（1）不管try中是否发生异常，也不管catch是否可以捕获异常，也无论try或catch中是否有return，finally中的代码都必须执行。
（2）如果finally中有return，就从finally块的的return回去。此时try和catch中的return语句执行了一半，执行了第一个动作。所以，finally中的return语句会覆盖刚刚的返回值。
（3）如果finally中没有return，那么先把try或catch中该执行的执行完，在return结束当前方法之前（此时return已经压栈不会被fially中的代码改变，类似i++的意思）先走一下finally，然后回去结束当前方法。

“return 返回值;”语句有两个动作：
（1）把返回值放到“操作数栈”中。
（2）结束当前方法的执行，把“操作数栈”中的值会返回给调用处。

5、异常对象的常用方法：
（1）e.printStackTrace();：打印异常的详细信息，包括追踪跟踪信息，即这个异常对象一路经过了哪些方法。
（2）e.getMessage();：返回异常对象中简单的错误信息提示。

6、打印异常/错误信息：
（1）System.err.println(xx);：打印错误信息，红字。
（2）System.out.println(xx);：打印正常信息，黑字。

7、面试题：finally,final,finalize的区别？
（1）finally：最终块
无论try中是否有异常，无论catch是否可以捕获异常，也不管try和catch中是否有return，都一定要执行的部分。
（2）final：是一个修饰符
修饰类：表示不能被继承。
修饰方法：表示不能被重写。
修饰变量：表示常量，值不能修改。
（3）finalize：是一个方法名，在Object类中声明的，当对象被GC回收之前调用。

throw/throws

Java异常类对象除在程序执行过程中出现异常时由系统自动生成并抛出，也可以根据需要手动创建并抛出。
首先要创建异常类对象，然后通过throw语句实现抛出操作，提交给Java运行环境。
可以抛出的异常必须是Throwable或其子类的实例。

语法格式：


throw：
如果在某个方法中，用throw语句手动抛出异常对象是运行时异常，方法可以不用加throws。
如果在某个方法中，用throw语句手动抛出异常对象是编译时异常，方法还要加try...catch或throws。
throw语句可以代替return语句结束方法，可以抛出“系统预定义异常”或“用户自定义异常”。

throws：
（1）告知被调用者，我这个方法可能会抛出哪些异常，使得调用者可以明确的知道应该catch什么异常，如果没有throws，那么调用者就不清楚，只能按照Exception处理，或者根据错误经验来处理。
（2）编译时的异常，如果在当前方法中不用try..catch处理，那么可以通过throws明确的说明，抛给调用者处理。
（3）方法重写时，对throws抛出的异常的要求：子类重写的方法抛出的异常类型必须<=父类被重写的方法抛出的异常类型。因为如果多态时按父类编译的异常<=按子类运行的异常，运行时会报错。

自定义异常：
在程序中，可能会遇到任何标准异常类都没能充分的描述清楚的问题，这种情况下可以创建自己的异常类。
（1）必须继承Throwable或它的子类，但是实际开发中，一般继承RuntimeException和Exception。
（2）建议大家保留两种构造器的形式，无参构造、带有详细信息（带给父类的message属性赋值）的构造器构造器。
（3）建议增加serialVersionUID，防止序列化和反序列不一致。
（4）自定义的异常只能通过throw抛出，它不能由JVM自动抛出。
（5）自定义异常最重要的是异常类的名字，当异常出现时，可以根据名字判断异常类型。
